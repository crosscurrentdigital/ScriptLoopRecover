Conduct a full production-readiness audit of this app. Inspect the codebase, run the app, trace all major flows, and evaluate whether it is actually ready for public release in production.

Do not stop at surface-level UI review. Audit the app across product, engineering, infrastructure, security, reliability, compliance, and release operations.

Your goal is to identify every meaningful issue, risk, gap, inconsistency, and missing piece that could prevent a safe, stable, scalable production launch, then provide a concrete remediation plan.

# Core Mandate

Audit this app as if you are the final internal reviewer before public launch.

You must evaluate:
- functionality
- UX and layout
- data flow and persistence
- auth and permissions
- error handling
- security
- privacy/compliance
- performance
- accessibility
- observability
- deployment/release readiness
- operational reliability
- legal/content/policy gaps
- support/admin readiness
- scalability and maintainability

Do not assume something works because the code looks correct. Verify by inspection, testing, tracing, and reasoning through edge cases.

If something cannot be verified directly, mark it explicitly as:
- Verified
- Likely OK but unverified
- Missing
- High-risk unknown

# Audit Scope

## 1) Product and Feature Completeness
Audit every user-facing and admin-facing feature.

Check:
- every page, modal, drawer, tab, menu, wizard, onboarding step, and settings area
- every button, link, form, input, dropdown, toggle, checkbox, radio, uploader, search bar, filter, sort, pagination control, and CTA
- empty states, loading states, success states, error states, and disabled states
- edge cases and unusual user behavior
- whether the implementation matches the intended product behavior
- whether important flows are incomplete, stubbed, or misleading

For each feature, determine:
- does it work
- does it fail gracefully
- does it produce correct data
- does it create confusing or dangerous states
- is it production-polished or still prototype-quality

## 2) End-to-End User Journey Audit
Trace all major user journeys from first touch to successful completion.

Include:
- first visit / landing
- sign up / sign in / sign out
- email verification / password reset / magic link / MFA if applicable
- onboarding
- core activation flow
- repeat usage flow
- upgrade / payment / subscription / checkout if applicable
- account management
- notifications / emails / confirmations
- cancellation / downgrade / renewal / billing portal if applicable
- support / help / contact / report issue
- logout and session expiration
- account deletion / data export if applicable

For each journey, identify:
- friction
- dead ends
- unclear copy
- hidden assumptions
- broken transitions
- trust issues
- abandonment risks

## 3) UI, Layout, and Design-System Audit
Inspect the entire interface for production-quality consistency.

Check:
- visual consistency across screens
- spacing, alignment, typography, colors, icon use, and component consistency
- responsive behavior on mobile, tablet, and desktop breakpoints
- dark mode/light mode if supported
- overflow, clipping, wrapping, truncation, z-index issues
- sticky headers/footers, navigation behavior, scrolling issues
- forms and validation UX
- modal focus traps and close behavior
- keyboard navigation
- whether components match the design system or are ad hoc

Call out:
- areas that feel unfinished
- inconsistent patterns
- places where the UI undermines trust or perceived quality

## 4) Functional Logic and State Management
Inspect how the app behaves internally.

Check:
- client-side state correctness
- server-side state correctness
- synchronization between UI state and backend state
- stale data risks
- race conditions
- double-submit risks
- optimistic updates
- cache invalidation
- retry behavior
- duplicate requests
- improper loading state handling
- navigation/state reset issues
- state leaks between users, tenants, or roles

## 5) Data Pipeline and Data Integrity
Trace data from input to storage to retrieval to display.

Audit:
- input validation
- sanitization
- transformations
- API requests/responses
- database writes and reads
- file uploads and storage
- background jobs / queues / webhooks if applicable
- derived fields and computed values
- reporting / analytics data correctness
- idempotency for retried operations
- duplicate record creation
- orphaned records
- soft delete / hard delete behavior
- migrations, seeds, and schema consistency

Identify:
- where bad data can enter
- where good data can be corrupted
- where data can be lost, duplicated, exposed, or become inconsistent

## 6) Auth, Roles, Permissions, and Tenant Boundaries

Check:
- sign-in flows

- server-side authorization enforcement
- role-based access control
- admin-only functions

- direct URL access to restricted resources
- hidden UI vs actual enforcement
- privilege escalation risks

Assume hostile usage and verify that users cannot access:
- other users’ data
- admin functionality
- restricted records
- cross-tenant resources

# Required Audit Method

You must:
1. Inspect the codebase structure and major modules.
2. Identify all routes/pages/screens and major components.
3. Identify all APIs, services, DB models, background jobs, and integrations.
4. Run and test the app where possible.
5. Trace critical user journeys end-to-end.
6. Trace data flows end-to-end.
7. Review environment/config/deployment files.
8. Review test coverage and missing tests.
9. Review dependencies and security-sensitive areas.
10. Produce a remediation plan prioritized for release.

Do not just summarize. Audit deeply and produce actionable findings.

# Severity Levels

For every issue, assign:
- Critical — launch blocker, security issue, data loss/exposure risk, broken core flow
- High — serious production risk, major UX break, compliance gap, reliability concern
- Medium — important but not launch-blocking
- Low — polish, maintainability, or minor UX issue

# Output Format

Produce the final output in this structure:

## 1. Executive Summary
- overall production readiness assessment
- top launch blockers
- top 10 risks
- go / limited beta / no-go recommendation

## 2. System Overview
- app type
- main user roles
- major flows
- key integrations
- architecture summary
- notable assumptions or unknowns

## 3. Findings by Category
For each category above, list:
- Issue ID
- Title
- Severity
- Area
- Affected files/components/routes/services
- Description
- Evidence
- Reproduction steps
- Expected behavior
- Actual behavior
- Root cause
- Recommended fix
- Effort estimate
- Owner type (frontend/backend/devops/design/product/security)
- Acceptance criteria

## 4. Production Release Blockers
A short list of issues that must be fixed before release.

## 5. Recommended Fix Plan
Organize fixes into:
- Phase 1: must fix before launch
- Phase 2: should fix immediately after launch
- Phase 3: hardening and scale improvements

For each phase include:
- tasks
- dependencies
- recommended order
- quick wins
- risk reduction impact

## 6. Missing Coverage / Unknowns
List anything that could not be fully verified and what should be checked manually.

## 7. Test Plan to Close Gaps
List the exact tests that should be added:
- unit tests
- integration tests
- end-to-end tests
- manual QA checks
- security tests
- performance/load tests

## 8. Final Go/No-Go Verdict
State clearly whether this app should be released to production right now and why.

# Standards

Be exhaustive, skeptical, and concrete.
Assume real users, real attackers, real scale, and real production consequences.
Do not provide vague advice.
Tie every recommendation to a specific issue, risk, or missing capability.
Prioritize release safety, trust, and correctness over optimism.
