# Database Migrations

ScriptLoop's schema changes are tracked as **versioned, committed SQL
files** under [`drizzle/`](./drizzle), generated by `drizzle-kit` from
[`src/db/schema.ts`](./src/db/schema.ts). The Netlify build runs
`drizzle-kit migrate` against the deploy target's `DATABASE_URL` before
the new code goes live, so a deploy can never land on a database that is
missing a column the new code expects.

## Why versioned migrations (and not `db:push`)

`drizzle-kit push` (a.k.a. `npm run db:push:dev`) compares your local
schema to whatever the database currently is, and quietly mutates the
database to match. That's convenient on a dev box where you own the DB,
but in production it has two failure modes we explicitly want to avoid:

1. **Silent destructive change.** A renamed column looks like "drop old +
   add new" to a diff tool — `db:push` will happily destroy data with no
   review.
2. **Unsafe rollback.** If you `git revert` a code change, `db:push`
   would re-diff against your old schema and mutate the DB *again*, with
   no audit trail.

Versioned migrations fix both: every change is a reviewed file, the
`__drizzle_migrations` table records what was applied and when, and a
rolled-back code deploy simply leaves the migration in place.

## Day-to-day workflow

1. Edit [`src/db/schema.ts`](./src/db/schema.ts) the way you would any
   other source file.
2. Generate a migration:
   ```
   pnpm --filter @workspace/scriptloop run db:generate -- --name=add_voice_label
   ```
   This writes a new SQL file under `drizzle/` and updates the
   `drizzle/meta/` snapshot.
3. **Read the SQL.** This is the single most important step — Drizzle
   sees a rename as a drop + add. Edit by hand if the diff is wrong.
4. Apply it locally to your dev DB (set `DATABASE_URL` to a dev branch):
   ```
   pnpm --filter @workspace/scriptloop run db:migrate
   ```
5. Commit `src/db/schema.ts`, the new SQL file, and the updated
   `drizzle/meta/` files together. The Netlify build will apply the
   migration to prod on the next deploy.

`pnpm --filter @workspace/scriptloop run db:check` runs `drizzle-kit
check` to surface common journal/snapshot drift before you commit.

## The expand / contract rule

**Every migration must keep the previously-deployed code working.** This
makes rolling back code safe (the schema is always at-least as new as
the code expects) and lets us deploy schema and code separately when
needed.

The pattern for any non-trivial change is **expand → migrate code →
contract**, applied across two or more deploys:

| Step | Schema | Code | Safe to roll back? |
| --- | --- | --- | --- |
| 1 (expand) | Add the new column / table / index. Old code ignores it. | Old code, unchanged. | Yes — old code never read the new column. |
| 2 (migrate code) | Unchanged. | Write to both old and new; read from new. | Yes — old data still readable from old column. |
| 3 (backfill) | Backfill historical rows into the new column. | Same as step 2. | Yes. |
| 4 (contract) | Drop the old column / constraint. | New code only. | Only after step 2 is fully rolled out. |

**Worked example — renaming `scripts.voice_id` to `scripts.voice_ref`:**

1. Migration: `ALTER TABLE scripts ADD COLUMN voice_ref text;`. Deploy.
2. Code change: write to both `voice_id` and `voice_ref`; read from
   `voice_ref` and fall back to `voice_id`. Deploy.
3. Backfill: `UPDATE scripts SET voice_ref = voice_id WHERE voice_ref IS
   NULL;`. Run as a one-off migration.
4. Code change: stop writing `voice_id`. Deploy.
5. Migration: `ALTER TABLE scripts DROP COLUMN voice_id;`. Deploy.

Steps 1–4 are individually rollback-safe. Only step 5 is a one-way door,
and by then you have weeks of confidence in the new column.

## Environments

Each environment has its own Neon branch and its own `DATABASE_URL`
secret:

| Environment | `DATABASE_URL` lives in | Schema applied via |
| --- | --- | --- |
| Local dev | `artifacts/scriptloop/.env` (gitignored) | `db:migrate` (or `db:push:dev` for throwaway iteration) |
| Production | Netlify → Site settings → Environment variables | `db:migrate`, automatically, as part of the Netlify build |

A staging Neon branch is not provisioned today; when it is, it slots
into this table the same way as production.

## The `neon_auth.user` table

The `neon_auth` schema and its `user` table are managed by Neon Auth,
not by us. The baseline migration declares the table with `CREATE TABLE
IF NOT EXISTS` only so the foreign key from `scripts.user_id` resolves
on a fresh database. Do **not** add migrations that alter
`neon_auth.user` — Neon Auth owns its lifecycle.

## When something goes wrong

- **Migration fails on deploy.** The Netlify build fails, no new code is
  served, the previous deploy keeps serving. Read the build log,
  fix the migration locally, push a new commit.
- **Migration applied but code is broken.** Roll back the code deploy in
  Netlify. Because every migration follows expand/contract, the
  previous code version still works against the migrated schema.
- **You ran `db:push:dev` against prod by accident.** Inspect with
  `db:check`; if drift exists, write a corrective migration that brings
  the schema back in line and commit it.
